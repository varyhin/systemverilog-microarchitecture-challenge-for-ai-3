## 1. Промт задачи

> Finish the code of a pipelined block in the file `challenge.sv`. The block computes a formula **`a ** 5 + 0.3 / b - c`**. Ready/valid handshakes for the arguments and the result follow the same rules as ready/valid in AXI Stream. When a block is not busy, `arg_rdy` should be 1, it should not wait for `arg_vld`.

Ограничения: не более 10 арифметических блоков, не более 10 000 D-триггеров вне арифметических блоков, без SRAM, синтезируемый RTL. Конвейер должен принимать данные без лишних пауз при отсутствии backpressure.

**Ключевое отличие от оригинала:** замена `0.3 * b` (фиксированная латентность, конвейеризованное умножение) на `0.3 / b` (переменная латентность ~16 тактов, делитель блокирует вход при busy=1). Это принципиально меняет архитектуру.

### 2.3. Анализ f_div (ключевой этап для версии с делением)

Детальный анализ `fdivsqrtfsm.sv`, `fdivsqrtcycles.sv`, `config-shared.vh`:

- **Алгоритм**: SRT radix-4 с DIVCOPIES=2 (4 бита/такт)
- **Итерации**: ceil(56/4) = **14 тактов** (NF=52 + 2 + LOGR=2 = 56 бит)
- **FSM**: IDLE → BUSY (14 тактов) → DONE (1 такт) → IDLE
- **Полная латентность**: ~16 тактов (от up_valid до down_valid)
- **Throughput**: 1 деление каждые ~16 тактов (НЕ конвейеризовано)
- **busy=1**: замораживает D/E регистры через StallE, новый вход невозможен

**Ключевой вывод**: один делитель даёт throughput ~1/16 — за 10000-тактовый timeout тестбенч не успеет обработать все входы. Нужно несколько делителей.

### 2.4. Расчёт необходимого числа делителей

| N делителей | Throughput | Тактов на 1000 входов | Влезает в TIMEOUT? |
|-------------|-----------|----------------------|-------------------|
| 1 | 1/16 | ~16000 | Нет |
| 3 | 3/16 | ~5333 | Пограничные |
| 4 | 4/16 | ~4000 | Да (впритык) |
| 5 | 5/16 | ~3200 | Да (с запасом) |

Бюджет блоков: 3 mult + N div + 1 add + 1 sub = 5 + N. При N=5: 10 блоков (на пределе).

**Решение:** 5 делителей round-robin.

### 2.5. Проектирование архитектуры с round-robin делителями

**Двухфазная архитектура:**

**Фаза 1 (front-end):** приём входов → распределение по 5 делителям round-robin
- `slot_wr_ptr` (0-4): указатель записи
- Каждый слот хранит (a, c) и принимает результат деления 0.3/b
- `div_start[i]` — комбинационный up_valid для выбранного делителя

**Фаза 2 (back-end):** когда деление завершено, подача в конвейер mult→add→sub
- Сбор результатов строго по порядку (`slot_rd_ptr`)
- 3 умножителя вычисляют a^5, далее сложение и вычитание
- Линии задержки: a (6 тактов), quotient (9 тактов), c (13 тактов)

**Фаза 3 (output):** FIFO глубиной 32 для backpressure.

### 2.6. Реализация в challenge.sv

Написан модуль, включающий:

1. **5 инстанций f_div** через `generate` — round-robin делители
2. **Контроллер слотов** — slot_wr_ptr, slot_rd_ptr, slot_active[], slot_done[]
3. **Захват результатов** — slot_quot[] заполняется по div_dv
4. **Сбор по порядку** — collect = slot_done[slot_rd_ptr]
5. **3 инстанции f_mult** — a², a⁴, a⁵
6. **f_add + f_sub** — a⁵ + quotient - c
7. **3 линии задержки** — a (6), quot (9), c (13) стадий
8. **Выходной FIFO** — 32 записи
9. **arg_rdy** — на основе свободности слота и заполненности FIFO

## 4. Ключевые решения и находки

1. **f_div радикально отличается от f_mult**: ~16 тактов переменной латентности, busy блокирует вход через `StallE = FDivBusyE`. Это определено из анализа `fdivsqrtfsm.sv` и `wally_fpu.sv`.

2. **Один делитель = timeout**: при throughput 1/16 тестбенч не успевает обработать ~2000 входов за 10000 тактов. Формула: N_div >= ceil(16/target_period). Для target ≈ 3.2: N=5.

3. **Round-robin 5 делителей**: каждый получает каждый 5-й вход. Все деления одинаковой длины (14 итераций для double), поэтому результаты приходят строго по порядку — не нужен reorder buffer.

4. **Двухфазная архитектура**: front-end (делители) и back-end (mult/add/sub) — развязаны через slot-контроллер. Back-end повторно использует паттерн из v1 решения (линии задержки + конвейер valid-сигналов).

5. **Комбинационный div_start**: up_valid для делителя должен быть в том же такте, что и входные данные b, иначе wally_fpu защёлкнет неверные операнды.

6. **10 блоков = предел бюджета**: ни одного блока в запасе, но DFF бюджет использован на 49%.
