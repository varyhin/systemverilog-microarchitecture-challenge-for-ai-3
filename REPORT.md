## 1. Промт задачи

> Finish the code of a pipelined block in the file `challenge.sv`. The block computes a formula **`a ** 5 + 0.3 * b - c`**. Ready/valid handshakes for the arguments and the result follow the same rules as ready/valid in AXI Stream. When a block is not busy, `arg_rdy` should be 1, it should not wait for `arg_vld`. You are not allowed to implement your own submodules or functions for the addition, subtraction, multiplication, division, comparison or getting the square root of floating-point numbers. For such operations you can only use the modules from the `arithmetic_block_wrappers` directory.

Ограничения: не более 10 арифметических блоков, не более 10 000 D-триггеров вне арифметических блоков, без SRAM, синтезируемый RTL. Конвейер должен принимать данные каждый такт back-to-back при отсутствии backpressure.

### 2.3. Определение точных латентностей

Ключевой этап — анализ `wally_fpu.sv` (обёртка) для определения числа тактов от входа до выхода.

Конвейер wally_fpu содержит 3 стадии регистров:
1. **D/E** (`flopenrc DEReg1..5`) — защёлкивает входные данные
2. **E/M** (`flopenrc EMFpReg1..7`) — защёлкивает промежуточные результаты FMA
3. **M/W** (`flopenrc MWRegFp, MWRegVld`) — защёлкивает финальный результат

Выходы `Res = FResultW` и `DownValid = VldW` берутся из стадии W.

Прослежена цепочка `UpValid → VldE (D/E reg) → VldM (E/M reg) → VldW (M/W reg) → DownValid`.

**Определены латентности:**

| Блок | Латентность | Обоснование |
|------|-------------|-------------|
| `f_mult` | **3 такта** | Напрямую подключен к wally_fpu (3 регистровые стадии) |
| `f_sub` | **3 такта** | Аналогично f_mult |
| `f_add` | **4 такта** | wally_fpu (3 стадии) + дополнительный `always_ff` выходной регистр (`res_r`, `down_valid_r`) |

**Результат:** точные латентности определены из кода, что критически важно для правильного выравнивания данных в конвейере.

### 2.4. Составление плана вычислений

Декомпозиция формулы `a^5 + 0.3*b - c`:

```
a^5 = (a*a) * (a*a) * a = a2 * a2 * a = a4 * a
```

Составлена таблица стадий конвейера:

```
Цикл 0:  Принять (a, b, c)
          mult1: a × a → a2                    (выход: цикл 3)
          mult2: 0.3 × b → p                   (выход: цикл 3)

Цикл 3:  mult3: a2 × a2 → a4                  (выход: цикл 6)

Цикл 6:  mult4: a4 × a_задержанное → a5        (выход: цикл 9)

Цикл 9:  add1:  a5 + p_задержанное → sum       (выход: цикл 13)

Цикл 13: sub1:  sum - c_задержанное → result   (выход: цикл 16)
```

Определены необходимые линии задержки:
- **a**: задержка 6 тактов (от цикла 0 до цикла 6 для mult4)
- **p** (результат 0.3×b): задержка 6 тактов (от цикла 3 до цикла 9 для add1)
- **c**: задержка 13 тактов (от цикла 0 до цикла 13 для sub1)

**Результат:** 6 арифметических блоков (4 mult + 1 add + 1 sub), полная латентность 16 тактов, ~1600 DFF на линии задержки.

### 2.5. Проектирование управления потоком (flow control)

Проблема: арифметические блоки не поддерживают стоп. Данные, попавшие в конвейер, выходят через 16 тактов неизбежно. При backpressure (`res_rdy=0`) результаты некуда отправлять.

Решение — **выходной FIFO**:
- Глубина 32 (степень двойки, >= глубины конвейера 16)
- Ширина 64 бита (FLEN)
- Реализация: кольцевой буфер с указателями записи/чтения

Управление `arg_rdy`:
- Ведётся счётчик **in-flight** операций (принятых, но ещё не дошедших до FIFO)
- `arg_rdy = ~rst & (fifo_count + in_flight < FIFO_DEPTH)` — принимаем данные, только если в FIFO гарантированно хватит места для всех in-flight результатов

Выходы:
- `res_vld = (fifo_count != 0)` — есть данные в FIFO
- `res = fifo_mem[rd_ptr]` — голова FIFO

**Результат:** надёжная схема flow control, ~2048 DFF на FIFO. Итого ~3700 DFF — в пределах лимита 10 000.

### 2.6. Реализация в challenge.sv

Написан полный модуль, включающий:

1. **Константы** — `CONST_0_3 = 64'h3FD3333333333333` (0.3 в IEEE 754 double), параметры FIFO
2. **Входное рукопожатие** — `accepted = arg_vld & arg_rdy`
3. **6 инстанций арифметических блоков** — mult1..mult4, add1, sub1 с правильной цепочкой `down_valid → up_valid`
4. **3 линии задержки** — сдвиговые регистры для a (6 стадий), p (6 стадий), c (13 стадий)
5. **Выходной FIFO** — кольцевой буфер на 32 записи с указателями `fifo_wr_ptr` / `fifo_rd_ptr`
6. **Счётчик in-flight** — инкремент при приёме, декремент при выходе из конвейера
7. **Логика arg_rdy** — на основе заполненности FIFO и числа in-flight операций

## 4. Ключевые решения и находки

1. **Латентность f_add отличается от f_mult/f_sub**: 4 такта вместо 3 из-за дополнительного выходного регистра в обёртке. Это определено путём чтения исходного кода, а не документации (которой нет).

2. **Арифметические блоки не стопятся**: конвейер wally_fpu всегда продвигает данные, нет механизма ready/stall для add/sub/mult. Это определило необходимость выходного FIFO.

3. **Оптимизация a^5**: использовано 3 умножения вместо 4 за счёт `a^5 = (a^2)^2 × a`.

4. **FIFO + in-flight counter**: обеспечивают корректное AXI Stream рукопожатие даже при произвольных паттернах backpressure.
